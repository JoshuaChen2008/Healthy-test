# PRD · 健康测评系统后端

> 面向：睿迄科技 3 天全栈挑战
> 本文档回答「要做什么」，不涉及实现细节。实现见 `tech-design.md`。

## 1. 背景与目标

做一个健康测评产品的**核心后端**（对标 BetterMe quiz funnel）：
用户分步填写身体数据 → 后端计算健康指标 → 结果页展示 →
非会员看到脱敏结果、付费后解锁完整数据。

UI 不是考查重点，能跑即可；后端工程质量与测试是核心。

## 2. 核心用户流程

```
开始测评 → 填性别 → 填目标 → 填身体数据(年龄/身高/体重/目标体重) → 填运动频率
  → 提交 → 后端计算(BMI / 建议摄入 / 目标日期) → 进入结果页
  → 非会员：脱敏结果 + 付费提示 → 点付费 → 解锁完整结果
```

关键约束：
- 用户中途关闭页面再进入，要能**恢复到上次填写的进度**。
- 同一用户可以**多次测评**（每次是一条独立记录）。
- 订阅是**账号级别**：付费后该用户所有测评都解锁。

## 3. 功能清单

| 编号 | 功能 | 输入 | 处理 | 输出 |
|------|------|------|------|------|
| F1 | 创建会话 | 无 | 生成随机 userId | userId |
| F2 | 新建测评 | userId | 建一条 in_progress 记录 | assessmentId |
| F3 | 分步保存 | assessmentId + 某步增量数据 | 校验 + 存库 + 更新当前步 | 保存成功 |
| F4 | 进度恢复 | assessmentId | 查该次测评已填数据 | 已填数据 + 当前进度 |
| F5 | 提交计算 | assessmentId | 校验完整性 + 算 BMI/摄入/目标日期 + 存结果 | 计算完成 |
| F6 | 结果返回 | assessmentId | 查该用户订阅状态，决定返回多少 | 会员=完整 / 非会员=脱敏 |
| F7 | 模拟支付 | userId | 订阅状态改为 active | 支付成功 |

## 4. 数据需求

- **要持久化**：用户身份、每步答案、当前进度、计算结果、订阅状态、支付时间。
- **被保护字段（非会员脱敏）**：`target_date`（目标预测日期）、以及任何"预测曲线"类数据。
- **默认状态**：新用户订阅状态 = `free`。

## 5. 非功能需求

- 刷新 / 中断后能恢复进度。
- 接口能挡住非法输入（负数、超范围、缺字段、非法类型、目标体重不合理等）。
- 核心逻辑有自动化测试，`npm test` 一键运行。
- 部署到公网，可完整演示 funnel + 模拟支付。
